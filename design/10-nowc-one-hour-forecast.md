# 修正設計: 気象庁ナウキャストに沿った「約1時間先まで」の予測フレーム統合

## このドキュメントで分かること

- **目的**: 防災気象の降水ナウキャスト（HRPNs）が **観測系（N1）と短期予報系（N2）に分かれている**事実に合わせ、ユーザーが **約1時間先までの予測タイル**も同一UIで確認できるようにするための**変更範囲と規約**。
- **前提**: 気象庁が公開している URL は **公式 API 仕様書としての保証は薄い**一方、コミュニティ検証によれば **N2 は同一 basetime に対し validtime をおおよそ +5〜+60 分程度まで並べる** JSON である（実際の可否は取得結果が唯一の根拠）。

設計方針は **追加フィールド中心の後方互換**、ロジックは **`domain` に集約**、データは **KV のバージョン付きスナップショット**でまとめる。

---

## 背景とゴール

### 現状（変更前）

- イングエストは **`targetTimes_N1.json` のみ**を KV に保存する。
- メタの `frames` はその一覧から構築するため、**短期予報コマ（N2）が一覧に現れない**。
- タイル URL パターン自体は **`basetime` / `validtime` の組で予報も取得可能**であり、プロキシ実装はすでに両者を受け付ける。

### ゴール（変更後）

- **気象庁が N2 で公開している時間軸（約1時間先までのコマ）**をシステムが扱えるようにする。
- ユーザーは **観測に近いコマと、短期予報コマを連続したタイムラインとして**閲覧できる（初期表示ルールは別途 `default_frame_id` の方針に従う）。
- **運用・保守**: 取得失敗時の劣化動作、KV マイグレーション、メタ契約の最小拡張を明文化する。

## 上流データの整理（規範ではなく設計上の前提）

| リソース | 役割（設計上の理解） | 本プロジェクトでの扱い |
|----------|----------------------|-------------------------|
| `.../targetTimes_N1.json` | いわゆる **実況／解析に近い時刻列**（多くの行で `basetime === validtime`） | 現状どおり取得。**観測系タイムライン**の主ソース |
| `.../targetTimes_N2.json` | **同一 basetime** を起点に **`validtime` だけが未来側へ進む**短期予報コマ（間隔はおおよそ 5 分前後） | **新規に取得**し、**予報系タイムライン**のソース |

同一のタイルパス規則を用いる（既存の `upstreamTileUrl` と整合）。

## アーキテクチャ上の変更（論理モジュール）

### 1. イングエスト（`radar-ingest` 相当: Worker 内）

**必須（MUST）**

- Cron（および非本番のウォームアップ）で **`N1` と `N2` の両方を HTTP GET** する。
- それぞれパースに成功した本文と、取得完了時刻を **`fetchedAtMs`（単一）**とともに KV に保存する。

**推奨（SHOULD）**

- 両取得は **`Promise.all`** で並列化し、タイムアウトは既存ポリシーに合わせる。
- **`N2` のみ失敗**した場合は **N1 のみでスナップショットを更新**し、メタで「予報が無い」ことをクライアントが判別できるようにする（後述の `forecast_available`）。

**禁止（MUST NOT）**

- N1 も N2 も失敗したときに **古い KV を無断で「成功」として見せない**（現行のエラー方針を踏襲。必要なら ingest をロールバックしないが、`stale`・ヘルスで検知可能にする）。

### 2. KV スナップショットの版上げ

**現状**: `v: 1` で `body`（N1 のみ）＋ `fetchedAtMs`。

**変更後**: `v: 2` を導入する。

```json
{
  "v": 2,
  "fetchedAtMs": 1710000000000,
  "n1Body": "[...]",
  "n2Body": "[...]",
  "n2FetchOk": true
}
```

**互換（MUST）**

- `readSnapshot` は **`v: 1` を読んだら従来どおり N1 のみ**として扱う（予報なし）。
- `v: 2` では `n1Body` / `n2Body` をパースし、**エントリ配列をマージ**する。

**任意（MAY）**

- `n2FetchOk: false` のとき N2 本文は空文字または省略し、メタに `forecast_available: false` を付す。

### 3. ドメイン（純粋関数）

**必須（MUST）**

- **フレーム ID**は現行どおり `basetime + "_" + validtime`。N1/N2 のいずれ由来でも衝突しなければ同一タイルキーになる。
- **`role`（観測寄り／予報寄り）**はクライアントが推測しないよう、サーバが決める。

**推奨ルール（SHOULD）** — 後から JMA 側表現が変わったときに差し替えやすいよう **1 関数に閉じる**:

- `basetime === validtime` なら `role = "analysis"`（実況・解析寄り）
- それ以外なら `role = "forecast"`（短期予報コマ）

※ N2 は通常 `validtime > basetime` のため `forecast` になる想定。

**ソートキー（MUST）**

- メタの `frames` は **`validtime` を UTC ISO に変換した文字列で昇順**（現状と同じ）。N1 と N2 を混ぜても **時系列が一意に並ぶ**。

**重複（SHOULD）**

- 同一 `(basetime, validtime)` が万一両方にあれば **1 件にマージ**（後勝ちまたは N1 優先など、実装で一貫した規則を一文で固定する）。

### 4. メタデータ契約（v1 のまま拡張）

`contract_version: "1"` を維持し、**任意フィールドのみ追加**する（既存クライアントは無視してよい）。

| フィールド | 型 | 意味 |
|------------|-----|------|
| `frames[].role` | `"analysis"` \| `"forecast"` \| 省略 | 当該フレームの性質。省略時は従来互換のためクライアントは「不明」としてよい |
| `forecast_available` | boolean | **短期予報コマがメタに含まれるか**（N2 取得失敗時 false） |
| `latest_analysis_time` | string \| null | **optional**。`latest_available_time` と役割が混ざる場合に分解するならこちらへ寄せる |

**互換注記**

- 既存の `latest_available_time` の意味が「最新の観測時刻」なのか「タイムライン最終」なのかが曖昧になるため、本変更では **どちらか一方に固定する規約を `02-system-architecture.md` に追記する**（推奨: **観測系の最新＝ N1 由来の最大 validtime**）。

### 5. タイル API

**変更なし（MUST NOT 改変）**

- URL 形はそのまま。`frame_id` に N2 の `(basetime, validtime)` が入れば、そのまま上流へプロキシ可能。

## フロント（Web）の振る舞い（設計レベル）

**最小（MUST のうち実装コストが低いもの）**

- `frames[].role` が付いたら、**現在のコマ表示ラベルに「観測／予報」**を出し分ける。
- `forecast_available === false` のとき、**予報が無い理由をユーザー向け短文で表示しない**（ステータス文言は任意）。ログだけでもよい。

**推奨（SHOULD）**

- タイムライン操作（前後コマ・再生）が **N1→N2 をまたいでも途切れない**こと（フレーム配列はサーバでソート済みのため、基本はそのまま動く）。
- `maxFrames`（現状 24）では **予報コマが末尾から欠ける**可能性があるため、**観測＋予報のバランス**を取るなら **上位ロジックを server 側で「直近観測 N 件＋予報 M 件」**に切るか、`maxFrames` を見直す（別タスクで数値決定）。

## 運用・監視

- **`/healthz`**: `last_ingest_ms` に加え、**`n2_ok`（boolean）**を返すと運用が楽（任意）。
- **ログ**: ingest 時に `n1_ok`, `n2_ok`, `merged_frame_count` を構造化ログに出す（SHOULD）。

## リスクと対応

| リスク | 対応 |
|--------|------|
| N2 の JSON 構造変更・欠落 | パース失敗時は N1 のみで劣化、`forecast_available: false` |
| 同一タイムラインが長くなり UI が煩雑 | ラベル・色分け・将来の「予報だけジャンプ」（任意） |
| `latest_available_time` の意味混線 | `02` に語義を一本化し、必要なら `latest_analysis_time` を追加 |

## 実装タスク一覧（チェックリスト）

# 設計書索引

## このドキュメントで分かること

- **設計書全体の読み方**：どの文書をどんな順で読むと迷わないか（おすすめの順番）。
- **用語の意味**：レーダー・タイル・coverage など、後の章で出てくる言葉をここでざっくり把握できる。
- **実装で先に決めておきたいポイント**：デプロイ先・メタデータの形・公開の順番など、後から変えると大変な「固定点」の一覧。
- **規範と参考の違い**：どこまでが「守るルール」で、どこが「例や補足」か。

## ドキュメント規約

- **規範（normative）**＝**守るべきルール**です。「しなければならない／してはならない」と書いた部分は、実装・運用で必ず従います。
- **参考（informative）**＝**読み物・例・背景説明**です。イメージを掴むためのもので、実装をそのまま縛るものではありません。

本文では RFC 2119 に沿った言い方を使います。意味は次のとおりです（括弧内は一言の補足）。

- **必須（MUST）**＝そうしなければならない
- **推奨（SHOULD）**＝そうした方がよい（例外は理由付きでよい）
- **任意（MAY）**＝そうしてもよい／しなくてもよい
- **禁止（MUST NOT）**＝そうしてはならない

## 読み順（推奨）

設計書は「最初から全部」ではなく、**作りたいことに近い順**で読むと理解しやすいです。次の順番は、**いま何を作るか → システムの形 → 公開の仕方**という流れになっています。

| 順番 | ファイル | ひとことで |

| 2 | `02-system-architecture.md` | **部品の役割**、**メタデータ JSON の約束**、**サーバの置き場所のイメージ** |
| 3 | `09-deployment-github-pages-and-cloudflare.md` | **規範どおりのデプロイ**（GitHub Pages ＋ Workers、CORS、`tile_url_template`） |
| 4 | `03-data-pipeline-and-timing.md` | **いつ・どの順でデータを出すか**、Cron、ポーリング |
| 5 | `04-module-boundaries.md` | **モジュールの役割**と、**依存の向き**（どこからどこを呼ぶか） |
| 6 | `05-non-functional-and-risks.md` | **運用・設定・秘密情報**、リスク |
| 7 | `06-minimal-stack-apis-and-services.md` | API の種類と、**最小構成のスタック**の補足 |

### 実装者が先に固定すること（設計上の固定点）

後から変えると手戻りが大きくなりやすい項目です。各項目の詳細は括弧内の設計書を参照してください。

- **GitHub Pages のオリジン**と **Workers API のオリジン**、および **CORS 許可リスト**（`09`）
- メタデータ API の **JSON スキーマ**・**`tile_url_template` は絶対 URL**・**契約バージョン**・**coverage**・エラー方針（`02`）
- **公開モデル**（メタデータ更新とタイル取得可能の順序、`T_pub` の意味、coverage 内タイルだけを公開保証対象にすること）（`03`）
- ポーリングの **基準（宣言周期）**・ジッター・最小間隔・429 時の扱い（`03`）
- **タイルキャッシュ層の使い分け**（Cache API を高頻度タイル応答、KV をメタ索引・短期状態、R2 は必要時のみ永続化）（`03`, `09`）

## 用語

後の章でそのまま使われる言葉です。**ひとことで**は日常語に近い説明、**意味**は設計上の定義です。

| 用語 | 意味 | ひとことで |
|------|------|------------|
| レーダー画素・グリッド | 気象当局が配信する反射強度などの格子データ | 雨の強さなどが並んだ「マス目」の元データ |
| タイル | Web 地図向けの XYZ / TMS 等の分割画像またはベクトルタイル | 地図を小さな画像の集まりに切った一枚一枚 |
| 正規化 | ソース固有フォーマットを内部共通表現へ変換すること | 出どころごとの形式を、アプリ内の共通フォーマットにそろえること |
| T_obs | 観測時刻（プロバイダ付与または推定。推定時はフラグを付与） | 「この雨のデータはいつの観測か」の時刻 |
| T_recv | イングエスト完了時刻 | サーバがデータを取り込み終わった時刻 |
| T_pub | フレームが「公開済み」と見なされる時刻（`03` の公開モデルで定義） | ユーザー向けに「このフレームは使える」とみなす時刻 |
| latest_available_time | メタデータが提示する、クライアントが既定で表示してよい最新フレームの観測時刻 | 「いま画面のデフォルトで見せてよい最新」の時刻 |
| stale | プロバイダ期待周期に対しデータが古い、または取得失敗が続いている等の理由で鮮度が欠ける状態 | データが期待より古い・途切れがち、という「鮮度の警告」 |
| coverage | rainmap が当該データセットでタイル取得を保証する地理範囲。範囲外タイルは成功保証対象外 | **保証する地図上の範囲**（この外は「取れなくても仕様どおり」） |
| covered tile | `coverage` と `zoom_range` の両方に含まれるタイル。公開モデルの「GET 成功」はこの範囲に限定する | 範囲もズームも条件を満たすタイル＝**成功を約束する対象** |

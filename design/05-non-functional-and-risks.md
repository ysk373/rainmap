# 非機能要件・保守性・リスク

## このドキュメントで分かること

- **設定や秘密情報をどう管理するか**（リポジトリに書くものと、クラウドだけに置くものの分け方）
- **運用で「何が起きているか」を追うためのログの考え方**（観測可能性・相関 ID・構造化ログの意味）
- **パフォーマンスやキャッシュの方針**（タイルが速く・安全に届くためのルール）
- **想定されるトラブル（リスク）と、設計上どう抑えるか**

初めて読む方は、まず「保守性」の **観測可能性・相関 ID** の説明ボックスを読み、そのあと表形式のリスク一覧を見ると全体像を掴みやすいです。

---

## 保守性

- **設定の単一ソースと優先順位（規範）**: リポジトリにコミット可能な **デフォルト YAML**（または `wrangler.toml`）を下敷きとし、**環境変数で上書き**する。同一キーが両方に存在する場合 **環境変数が勝つ（MUST）**。**フロント用の API ベース URL**は GitHub Actions のビルド時変数、**秘密**は Cloudflare のみ、と役割を分ける（`09`）。

### 観測可能性（初心者向け）

**観測可能性**とは、「システムの中で何が起きたかを、後からログなどでたどれるようにしておくこと」です。バグ調査や「遅い・壊れた」の原因切り分けに使います。

**構造化ログ**とは、ログを人間向けの長い文章ではなく、**項目名と値がはっきりした形（JSON など）**で残すことです。機械的に検索・集計しやすくなります。例として次のような項目を想定する（規範として列挙する観測の例）:

| 項目（例） | ざっくり意味 |
|------------|----------------|
| `frame_id` | どの観測フレーム（時刻の塊）の処理か |
| `z/x/y` | どの地図タイル座標か |
| `coverage_hit` | データが「範囲内」だったかどうか |
| `cache_status` | キャッシュに当たったか・外れたか |
| `latency_ms` | どれくらい時間がかかったか（ミリ秒） |
| `outcome` | 成功・失敗など、結果の種類 |

**相関 ID（規範）**: バックグラウンドの一連の処理を「同じ仕事」としてつなぐための **共通の識別子**です。スケジューラがジョブ開始時に `correlation_id` を発行し、**フェッチ→検証→正規化／レンダ→ストレージ→（任意）メタ更新**の各ログに **同一 ID を付与（MUST）**。こうすると、ログを時系列で並べ替えなくても「このジョブの流れ」をまとめて追えます。タイル同期リクエストパスでは、リクエスト ID を別途付与してよいが、**オンデマンド生成がバックグラウンドジョブに接続する場合**はジョブの `correlation_id` へリンクするフィールドを残す（SHOULD）。

- **高ボリュームパス**: タイル GET のログは **サンプリング**してよい（例: 1%）。エラーとキャッシュミスは **常時 FULL ログ（SHOULD）**。（タイル取得は回数が膨大になりやすいので、全部を詳細ログにするとコストとノイズが増える。その代わり、困りごとは残す、というバランスです。）
- **契約テスト（規範）**: プロバイダ adapter のフィクスチャ応答に対し、**正規化後のドメインオブジェクト**（格子の形状・値域マッピング・時刻）までスナップショット検証する。HTTP のみのテストに留めない。
- **ログに載せないもの（MUST NOT）**: プロバイダレスポンス**本文**、URL クエリに含まれる可能性のある**鍵**、ユーザー個人を特定しうる入力。

## GitHub Pages と Cloudflare の設定分担（規範）

- **フロント公開情報**: API のベース URL（例: `https://rainmap-api.example.com`）は **ビルド時に環境変数で注入**する。値はパブリックでよいが、**リポジトリに上流秘密を混ぜない**。
- **秘密・本番ガード**: **Wrangler `secret` / Cloudflare ダッシュボード**にのみ置く（`09`）。
- **CORS**: Workers が **GitHub Pages のオリジン**を許可リストで返す（`09`）。

## シンプル性（意図的な削ぎ落とし）

- 初期は **1 プロバイダ活性 + フェイク** に限定
- 認証・RBAC は載せない
- マルチリージョン冗長は設計上の「将来」枠に留める

## 処理タイミング・性能

- 同期パス（ユーザーのタイルリクエスト）では **重い再投影を避け**、キャッシュヒットを第一にする
- **タイル応答の最適化（規範）**: フレーム ID を含むタイルはイミュータブルに近い `Cache-Control` を付与し、Cloudflare の Cache API / エッジキャッシュを主経路にする。KV はメタ索引・状態に限定し、高頻度タイルバイト列の恒常保存先にしない（`03`, `09`）。
- **範囲外リクエストの抑制**: クライアントはメタデータの `coverage.bbox` でオーバーレイ範囲を制限し、サーバは範囲外タイルを短い TTL の `404` または透明タイル等で軽く返す（`02`）。
- バックグラウンドで **失敗しても全体を落とさない**（デグレード運用）

## リスクと緩和

| リスク | 緩和 | 初心者向けメモ（ユーザー視点で困ること） |
|--------|------|------------------------------------------|
| プロバイダ仕様変更 | adapter 分離、バージョン付きフィクスチャテスト | 気象データ元の形式や URL が変わると、**地図に映らない・ズレる**ことが起きやすい。 |
| レート制限 | バックオフ、共有キャッシュ、条件付き GET | 取りすぎると **403/429 などで取得停止**になり、ユーザーには「更新が止まった」ように見える。 |
| 法務・利用規約 | 利用条件の記載、属性表示義務を UI に組み込む（実装フェーズ） | 違反すると **サービス停止や法的リスク**。クレジット表示などはユーザー体験と直結する。 |
| 誤った座標投影 | domain で変換パラメータを明示、サンプル BBox のゴールデンテスト | 投影がずれると **レーダーと地図が噛み合わない**（見た目の信頼が失われる）。 |
| **本番でフェイクプロバイダが有効化** | `01` の MUST NOT ガード、デプロイチェックリスト、設定スキーマでの列挙型 | テスト用データが本番に出ると **実データではない表示**になり、重大な誤情報になる。 |
| **CORS 未設定・許可オリジン誤り** | `09` に従い GitHub Pages の確定オリジンを列挙。カスタムドメイン変更時に追随 | ブラウザが API をブロックし、**画面は出るがデータが取れない**典型的な不具合になる。 |
| **タイルバイト列を KV に寄せすぎる** | Cache API を主経路、KV は小さな索引・状態、永続化が必要なら R2 を検討（`03`, `09`） | コスト・遅延・制限に当たり、**タイル表示が不安定や遅い**原因になりうる。 |
| **coverage 未定義による範囲外タイル急増** | メタデータに `coverage` を必須化し、UI とサーバ双方で範囲外リクエストを抑制（`02`） | 無駄なリクエストが増え **遅い・サーバ負荷・プロバイダ制限**に直結する。 |

### ゴールデンテストの例（informative）

固定した小さな入力ラスタと投影パラメータに対し、既知の角ピクセル（左上・右下）の値と、対応する WebMercator タイル 1 枚のハッシュが一致することを検証する。

## レビュー反映（rev.2）

- 「単一設定ファイル」と環境変数の二言説明だったため、**優先順位**を明記: *デフォルト YAML（コミット可）＜環境変数オーバーライド*。
- ログに **プロバイダレスポンス本文は載せない**（PII／鍵混入リスク）。メタのみ。

## レビュー反映（rev.3）

- **相関 ID の伝播**とタイルログの **サンプリング**を追記。
- 契約テストの到達点を **domain 出力**まで含めると明記。
- **設定ミス（フェイク本番）** をリスク表に追加し `01` と整合。

## デプロイ反映（rev.7）

- GitHub Pages と Workers の **設定分担・CORS** を追記（`09` と整合）。

## 最適化反映（rev.8）

- タイル応答のキャッシュ層を **Cache API / エッジキャッシュ優先**へ明確化し、KV 依存をメタ索引・状態中心に限定した。
- `coverage` による範囲外リクエスト抑制と、その観測項目を追加した。

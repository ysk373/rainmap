## このドキュメントで分かること

- **rainmap が「どこからデータを取り」「何をブラウザに公開するか」** の整理（上流 API・自前 API・地図 API の違い）

本書は API の区分と、上記方針におけるサービス役割を整理する。

## 三種類の API のイメージ（初心者向け）

ブラウザとサーバのまわりには、名前が似ていて紛らわしい **3 種類の「API」** が出てきます。役割を分けると設計が読みやすくなります。

```
        [ 気象データなどの提供者 ]
                 │
                 │  上流 API（サーバが pull）
                 ▼
        ┌─────────────────────┐
        │ Cloudflare Workers   │  ← 自前 API（公開面）
        │ （rainmap バックエンド）│     ブラウザが HTTPS で呼ぶ
        └─────────────────────┘
                 ▲
                 │  メタ・タイル取得など
                 │
        ┌─────────────────────┐
        │  ユーザーのブラウザ   │
        └─────────────────────┘
                 │
                 │  ベース地図のタイルなど
                 ▼
        [ 第三者の地図 API ]
        （OSM 等。地図タイルの提供者）
```

- **上流 API**: rainmap の **裏側（Workers）が向き合う**データ取得先。ユーザーは直接呼ばないことが多い。
- **自前 API（公開面）**: **ブラウザが rainmap 用に呼ぶ** HTTPS API（この設計では Workers）。
- **第三者の地図 API**: **地図の下地**（道路・国名など）を表示するため、多くの場合 **ブラウザが別ホストから直接**タイル等を取得する。利用規約・API キーはプロバイダ次第。

---

## 1. API の区分（用語）

| 区分 | 説明 | 例 |
|------|------|-----|
| **上流 API** | 当システムが **pull** するデータ取得先 | 気象レーダー画像・タイル・格子データの HTTP(S) エンドポイント |
| **自前 API（公開面）** | ブラウザが呼ぶ **HTTPS API**（本プロジェクトでは **Cloudflare Workers のオリジン**） | メタデータ JSON、タイル GET、ヘルス |
| **第三者の地図 API** | ベースマップ表示のためクライアントが取得するタイル等 | OSM 公式タイル、または別プロバイダのベクトル／ラスタタイル（利用規約・キーはプロバイダ依存） |

※ 「上流」の具体 URL は **利用規約・再配布可否を確認したうえで実装時に固定**する。設計では **`RadarProvider` アダプタの背後**として抽象化する。

---

## 2. 上流 API（データソース）の型（実装時にどれか選ぶ）

最小構成では **次のいずれか 1 系統**から開始する想定（複数ソース合成は後回し）。

| 型 | 概要 | 当システム側の仕事 |
|----|------|---------------------|
| **タイル URL 型** | 既に XYZ/WebMercator 等のタイル URL が並んでいる | フェッチして **`TileStore` に載せ替え／キャッシュ**（パススルーに近い） |
| **単時刻画像型** | PNG/WebP 等の広域画像 1 枚＋ジオ参照情報 | **切り抜き／再投影／タイル分割**が必要になりやすい |
| **格子データ API 型** | JSON／バイナリグリッド | **デコード・正規化・レンダ**が必要 |

開発・CI では **フェイク上流**（静的ファイルまたはモックサーバ）でよい（本番での有効化は `01-requirements-and-scope.md` の MUST NOT に従う）。

## 3. 自前 API（公開面）— 最小セット

ブラウザから見える **rainmap の HTTP API** は **Cloudflare Workers** で提供する。

| エンドポイント（論理） | 役割 |
|------------------------|------|
| `GET /api/v1/radar/meta`（例） | メタデータ JSON（契約は `02`。`coverage` と `{frame_id}` ベースの `tile_url_template` を含む） |
| `GET /tiles/...`（テンプレートはメタが指示） | タイルバイナリ |
| `GET /healthz`（例） | 死活・簡易依存チェック |

**オリジン分割（規範）**: フロントは **GitHub Pages**、API は **別オリジンの Workers**。そのため **`tile_url_template` は絶対 URL**、`Access-Control-Allow-Origin` は **`09`** に従いフロントオリジンを許可する。

---

## 4. サービス役割（本プロジェクト）

| サービス | 役割 |
|----------|------|
| **GitHub Pages** | SPA の静的配信のみ |
| **Cloudflare Workers** | メタ・タイル・上流フェッチ・（論理の）ワーカー／スケジューラ相当 |
| **Workers KV / Cron / Cache API** | KV はメタ索引・短期状態、Cron は定期ジョブ、Cache API は高頻度タイルキャッシュ（詳細は `09`） |

**Vercel・Railway・自前 VPS** は MVP の規範構成では使わない（無料枠・運用の都合で **将来の代替**としてよい）。比較・無料枠の話は `08-free-tier-hosting-options.md`。

---

## 5. Vercel を規範に含めない理由（informative）

サーバレスのみで **タイル大量 GET** と **長めの Cron 駆動パイプライン**を載せると、実行時間・コールドスタート・帯域で不利になりやすい。**エッジ＋キャッシュ**が効く Cloudflare Workers と、静的ホストに慣れた GitHub Pages の組み合わせを優先する。

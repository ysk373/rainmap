# 無料枠とホスティングの比較（参考）

## このドキュメントで分かること

- **このプロジェクトが「公式に」採用するホスティング**（GitHub Pages ＋ Cloudflare Workers）のざっくりした理由
- **フロントと API を分けると CORS やビルド設定が増える**が、なぜそれでも Workers 側に API を置くのか（負荷・キャッシュ・定期処理）
- **他にありうる構成**（Cloudflare Pages に寄せる、Vercel、Pages のみ）を、**参考**として比較する表

---

**規範のデプロイ**は **`09-deployment-github-pages-and-cloudflare.md`**（GitHub Pages ＋ Cloudflare Workers）とする。本書は **なぜその分割か**・**他に何があるか**を整理する参考資料である。各サービスの無料枠は**変更されるため、公開前に公式の現行プランを確認すること**。

### データ取得と CORS（この文書の前提のおさらい）

地図のタイルは**リクエスト数が非常に多い**ため、気象庁などへの直接アクセスは **Worker などのバックエンドに集約し、キャッシュする**方針です（詳細は `07`）。そのうえで、**Web ページは GitHub Pages**、**API は別ドメイン（Workers）** になるため、ブラウザから API を呼ぶとき **CORS**（別サイトからの取得をサーバーが許可する仕組み）の設定が必要になります（`09`）。簡単に言うと、「フロントの URL」と「API の URL」が違うとき、**API が『このフロントからなら OK』と応答ヘッダで示す**必要がある、という話です。

---

## 採用スタック（規範の要約）

次の表は、MVP で **こう置く** と決めている構成の要約です。

| 層 | 選定 | 主な理由 |
|----|------|----------|
| フロント | **GitHub Pages** | 静的ホスティングに慣れた運用に合う。 |
| API・キャッシュ・Cron | **Cloudflare Workers**（＋KV 等） | タイルはリクエスト数が多く、**エッジキャッシュ**と **Cron** が扱いやすい。 |

**トレードオフ**: **CORS** と **API ベース URL のビルド時注入**が必要（`09`）。

---

## 参考: 代替案 A — Cloudflare Pages にフロントも集約

**Cloudflare Pages** に SPA を載せ、**Functions** または同一ゾーンの Worker で API を配る。

- **利点**: 同一ドメイン配下にルーティングしやすく、**CORS が減る**。
- **欠点**: GitHub Pages 運用とリポジトリ分割・デプロイ導線が変わる。

本プロジェクトでは **GitHub Pages を維持**するため、規範からは外す（informative）。

---

## 参考: 代替案 B — Vercel 単体（または静的＋ Route Handler）

Next.js 等でフロントと API を一元化できるが、**タイルパンでの関数ヒット**と **Cron／実行時間**が無料枠で窮しやすい。チーム都合で Vercel に寄せる場合は **CDN キャッシュ設計を必須**とする。

---

## 参考: 「GitHub Pages のみ」（レーダープロキシなし）

静的ホストのみでは **メタ・プロキシ・定期取得が実装できない**。ブラウザが上流を直叩きする構成は **規約・負荷・CORS** の観点から本プロジェクトでは採用しない（`07`）。

---

## LINE Bot との組み合わせ

LINE から開く URL は **GitHub Pages の HTTPS** でよい（クエリで緯度経度・ズーム）。読み込み後の API 呼び出しは **Workers オリジン**へ向く（`07`, `09`）。

# データパイプラインと処理タイミング

## このドキュメントで分かること

- レーダーデータが「観測されてから、ユーザーに見せてよい状態になるまで」の**3つの時刻**（T_obs / T_recv / T_pub）の意味
- **公開モデル**（いつ・どの順でストレージ・タイル・メタデータを揃えるか）と、キャッシュ・ウォームアップの**規範（MUST 等）**
- **パイプラインのステージ**（ざっくり流れと詳細）と、スケジューラとフェッチを分ける**ジョブモデル**
- **ポーリング間隔・バックオフ**など、処理タイミングの方針表

---

## タイムモデル（3つの「いつ」）

レーダー画像が流れてくるとき、「いつのデータか」をきちんと区別します。次の3つを覚えると読みやすいです。

| 記号 | ひとこと | もう少し具体的に |
|------|----------|------------------|
| **T_obs**（観測時刻） | 「**現場で雨が観測された（とみなす）時刻**」 | データ提供者（プロバイダ）が付ける時刻。分からないときは、取得レスポンスのヘッダや規約に従って推定し、**推定した**ことはメタデータに**推定フラグ**として載せる。 |
| **T_recv**（到着時刻） | 「**うちのサーバがデータを受け取り終わった時刻**」 | イングエスト（取り込み）が完了した時刻。ネットワーク遅延などで T_obs より遅くなるのは普通。 |
| **T_pub**（公開時刻） | 「**ユーザーに見せてよい状態になった時刻**」 | 下記の**公開モデル**を満たし、フレームが「公開済み」とみなされる時刻。具体的には、メタデータがそのフレームを **`frames` に含み**、かつ **`coverage` と `zoom_range` で約束した範囲**についてタイル GET が成功する状態になったとき。 |

**設計上の指標**: **公開遅延** = T_pub − T_obs（「観測から、見せられるようになるまでどれだけ空いたか」）。SLA は実装後に計測して定義する。

---

## 公開モデル（規範）

クライアントから見ると、「メタデータには載っているのにタイルが取れない」などの**矛盾はゼロ**にします（`01` の成功基準と整合）。  
**単一フレームの公開は、データベースのトランザクションのように「一瞬で全部終わる原子操作」ではない**一方で、**ユーザーが観測してよい不整合は出さない**、という約束です。

1. **ストレージコミット**: 正規化済みラスタ、またはプロバイダタイルのパススルー実体が `TileStore` に書き込まれ、**一覧可能性**がある（部分書き込みのまま公開しない）。
2. **タイル可視化**: そのフレームについて、`coverage` と `frames[].zoom_range` で宣言した範囲のタイルが、オンデマンドまたはプリレンダーのいずれかにより **GET 成功可能**な状態にする。これは全 `x/y` を事前列挙・生成する意味ではなく、上流フレームの存在確認、座標変換、キャッシュキー生成、無降水時の透明タイル等への変換方針が確定しており、対象タイル要求を成功レスポンスへ決定できる状態を指す。**宣言ズームのサブセットだけ先行公開してはならない（MUST NOT）**。実装上どうしても段階が必要な場合は、メタデータに**より狭い `zoom_range` または `coverage` を載せてから**後で拡張する（メタデータが真実）。
3. **メタデータ更新**: ワーカーが **`radar-catalog` のユースケース**（公開記録）を呼び出し、索引を更新する。これにより当該フレームを `frames` に含む応答が可能になる。
4. **CDN 無効化**: CDN を使う場合、メタデータ更新後に古いパスが残っていても **フレームごとにパスまたは `id` が一意**であればキャッシュ汚染を避けられる（下記キャッシュキー規約）。

**`T_pub` の定義**: 上記 3 が満たされた時刻（オペレーション上は「メタデータがフレームを返した最初の応答の時刻」を T_pub の近似として計測してよい）。

---

## キャッシュキー規約（規範）

- タイル URL は **`tile_url_template` の `{frame_id}` にフレーム `id` を埋め込む**ため、フレームごとに **ネームスペースが分離**されること（MUST）。同一 `id` で内容が差し替わらないようにする（イミュータブルコンテンツとして扱う）。
- メタデータは **`contract_version` とデータセット世代**（実装で `dataset_revision` 等を追加してよい）で変えうるが、v1 では **`frames[].id` のみで十分**とする。

---

## キャッシュ層の使い分け（規範）

- **高頻度のタイル応答**は **Cloudflare Cache API / エッジキャッシュ**を第一キャッシュとする（MUST）。タイルバイト列を Workers KV に大量保存する設計を MVP の既定にしてはならない（SHOULD NOT）。
- **Workers KV** は、メタ索引、最終成功フェッチ時刻、連続失敗カウンタ、短期の欠損・負荷抑制フラグなど、小さく低頻度更新の状態に限定する（SHOULD）。
- **R2** は、上流再取得を避ける必要がある、または保持期間を長くする段階で導入する将来オプションとする（MAY）。MVP では Cache API と条件付き GET を優先し、永続タイルストアを必須にしない。
- `404` 等の欠損応答をキャッシュする場合は、フレーム公開直後の一時欠損を固定化しないよう **短い TTL** とする（SHOULD）。

---

## ウォームアップとオンデマンドの優先順位（規範）

- **既定**: タイルは **オンデマンド生成＋短期キャッシュ**。
- **ウォームアップ（任意・フラグで ON）**: 新規フレームについて、`coverage` 内かつ `zoom_range` のうち **代表 z（例: min〜min+2 または設定で指定）**の代表タイルだけを、メタデータ公開**前**にプリフェッチしてキャッシュを温める（SHOULD）。全 `x/y` の列挙ウォームアップは MVP では行わない（SHOULD NOT）。**メタデータにフレームを載せる前に**、宣言した `coverage` と `zoom_range` 全体が成功可能であることは変わらない（公開モデル MUST）。
- ユーザーリクエストによるオンデマンド生成とウォームアップが競合した場合、**同一キーの書き込みは冪等**とみなし、重複生成を許容するが、**Externally visible な「未完成タイル」レスポンスを返してはならない（MUST NOT）**（生成中は 504 またはバックグラウンド完了待ちなど、実装フェーズで選択）。

## パイプライン・ステージ

### ざっくり流れ（全体のイメージ）

1. **予定どおり起動** — スケジューラが「いま誰を取りに行くか」を決める（実際の取得はしない）。
2. **取り込み** — ネットから生データを取る → 壊れてないか確認 → 内部形式にそろえる（必要なら）。
3. **保存とタイル** — 状態は KV、タイルの速い応答はエッジキャッシュ中心。必要なら将来 R2。
4. **地図用タイルにする** — 宣言したズーム範囲に沿ってタイル化（オンデマンド／ウォームアップは上の規範どおり）。
5. **公開** — カタログに載せて、クライアントがメタデータでそのフレームを知れるようにする。

### ステージ一覧（詳細）

1. **スケジューラ** — プロバイダの更新周期に合わせてジョブを起動。**本プロジェクトでは Cloudflare Cron Triggers** がこの役割を担う（`09`）。
2. **フェッチ** — 生データ取得、HTTP キャッシュヘッダ尊重、条件付き GET（ETag / Last-Modified）
3. **検証** — サイズ・CRC／マジックバイト・座標メタの妥当性
4. **正規化** — 内部共通ラスタ（投影・解像度・値域の映射）。パススルーの場合はスキップし adapter が `TileStore` へ配置（`04`）
5. **永続化／キャッシュ** — メタ索引や状態は **Workers KV**、高頻度タイル応答は **Cache API / エッジキャッシュ**を主とする。タイルバイト列の永続化が必要になった場合のみ **R2** を検討し、従来の「単一ボリューム」に相当する配置は `09` に従う
6. **タイル化** — 必要なズームレンジはメタデータの `zoom_range` と一致させる。オンデマンド／ウォームアップは上記優先順位に従う
7. **公開** — ワーカーが **`radar-catalog` のユースケース**を呼び、公開済みフレームを索引に記録したうえで、メタデータ応答が当該フレームを返せる状態にする（CDN／エッジキャッシュはパス設計と `Cache-Control` で制御。**本プロジェクトでは Cloudflare、`09`**）

---

## ジョブモデル: スケジューラとフェッチの分離（規範）

- **スケジューラ**は「いつどのプロバイダを対象にジョブを発行するか」のみを決定する。**I/O を直接行ってはならない（MUST NOT）**。
- **ジョブキー**: `provider_id` + `logical_frame_key`（プロバイダが返す版・時刻・ハッシュ等）で **冪等**とする。同一キーの並行実行は **重複抑止**（ロックまたは「実行中」状態）する（MUST）。
- **再試行境界**: フェッチ失敗 → フェッチから再試行。検証失敗 → 破棄しアラート。正規化失敗 → 正規化から（入力ラスタが同一なら同一ジョブキー）。タイル化失敗 → タイル化から。メタデータ公開前に **部分状態を外部へ見せない**。
- **バックプレッシャー**: ワーカーキュー長に上限を設け、超過時は**ドロップではなく遅延実行またはスケジューラ側で間引き**する（具体アルゴリズムは実装フェーズ）。いずれにせよ **プロバイダへのリクエストレート**は下記ポーリング規則を超えない。

## 処理タイミング方針

| 項目 | 方針 |
|------|------|
| ポーリング基準 | プロバイダが文書化する**宣言更新周期** `T_decl` を正とする。観測により実効周期が分かる場合はメトリクスで `T_decl` を校正してよいが、**初期実装は `T_decl` のみ**でよい（シンプルさ優先） |
| ポーリング間隔 | **基本間隔 = `T_decl`**。全インスタンスで同一周期になるとスパイクするため **±10% の一様ジッター**を付与する（MUST）。**最小間隔 `T_min`**（例: 30s）は設定で固定し、それ以下にはしない |
| 過負荷時 | HTTP **429** または **Retry-After** がある場合は **それに従う（MUST）**。無い場合は指数バックオフし、最大間隔に上限を設ける |
| バックオフ | 連続失敗で指数バックオフ、最大間隔に上限 |
| タイル生成 | **オンデマンド既定**＋任意ウォームアップ（上記） |
| 古いデータ | TTL に達したフレームは削除ポリシー（例: 過去 6 時間）。削除後は次回メタから除外 |

**つまり**: 「いつ問い合わせるか」はプロバイダの宣言周期を基準にし、**同時アクセスの山を避けるジッター**と、**429 や Retry-After を無視しない**ことが規範として押さえどころ。

※ 旧案の「更新間隔の 0.5〜1 倍」は基準が曖昧なため **廃止**した。

## 欠損・遅延時の UX 契約

- メタデータで **latest_available_time** と **stale** を返す
- クライアントは最終フレームが古い場合でも地図は表示し、バナーで通知（実装フェーズ）

以下は設計の**変更履歴**（レビュー・デプロイ・最適化の反映）です。

**要約**: 公開モデル・キャッシュ・ジョブ・ポーリングを段階的に具体化し、Cloudflare（Cron・Cache API 等）に寄せた記述が追加されています。

## レビュー反映（rev.2）

- 「オンデマンドのみ」だと初回ヒットで遅くなるため、**ズーム境界の代表レベル（例 z=6〜8）のみウォームアッププリフェッチ**するオプションを設計に追加（フラグで無効化可能）。
- スケジューラとフェッチを同一ループに詰め込まない。**スケジュール決定**と**I/O 実行**をインターフェース分離（責務・テスト容易性）。

## レビュー反映（rev.3）

- **公開モデル**・**T_pub**・メタ／タイルの整合を規範化。
- **キャッシュキー**・ウォームアップとオンデマンドの順序を固定。
- **ジョブの冪等性・重複抑止・再試行境界**を追記。
- ポーリングを **`T_decl` + ジッター + 最小間隔 + 429** に具体化。

## デプロイ反映（rev.7）

- スケジューラを **Cloudflare Cron Triggers** に対応付け（上記ステージ 1）。

## 最適化反映（rev.8）

- 公開保証対象を **`coverage` と `zoom_range` の交差範囲**に限定し、全世界・全タイルの事前生成を暗黙要件にしないよう公開モデルを整理した。
- タイルキャッシュは **Cache API / エッジキャッシュ優先**、KV はメタ索引・短期状態中心、R2 は将来オプションとして使い分けを明確化した。
- ウォームアップは代表タイルに限定し、MVP での全 `x/y` プリフェッチを避ける方針にした。

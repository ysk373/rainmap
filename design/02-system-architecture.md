# システムアーキテクチャ

## このドキュメントで分かること

- **システムのブロック図**：外部の雨データから、ブラウザの地図まで、データがどう流れるか。
- **どこに何を置くか（規範）**：GitHub Pages と Cloudflare Workers の役割、CORS、`tile_url_template` が絶対 URL である理由。
- **メタデータ API の約束**：フロントがサーバから受け取る JSON に、どんな項目が必須か（契約 v1）。

## 論理構成と物理配置

### 論理レイヤ

```
[外部 レーダープロバイダ] 
        ↓ 取得（pull）
[イングエスト／正規化]  ←── Cloudflare Workers 上で実装（`09`）
        ↓ 内部ラスタ／メタデータ
[タイル／画像レンダリング]（任意: ソースがタイルなら薄くできる）
        ↓ API 応答（Cache-Control によりエッジキャッシュ可）
[クライアント地図 UI]  ←── GitHub Pages（静的）
```

**ひとことで**：気象側からデータを引き、サーバ側で整えてタイル化し、ブラウザは API 経由でタイルと説明（メタデータ）を受け取る、という流れです。

### 物理配置（規範）

MVP の **配置**は **`09-deployment-github-pages-and-cloudflare.md`** に従う（要約）:

- **フロント**: GitHub Pages（静的 SPA のみ）
- **API**（メタ・タイル・ヘルス・定期ジョブ）: **Cloudflare Workers**（＋KV / Cron 等）

フロントと API は **別オリジン**（URL の「ホスト部分」が違う）になるため、メタデータの **`tile_url_template`** は **API オリジン上の絶対 HTTPS URL** とする（MUST＝そうしなければならない）。ルート相対パスに依存してはならない（MUST NOT＝そうしてはならない）。

**CORS**: API（Workers）がフロントの GitHub Pages オリジンを **明示的に許可**する（`09`）。  
**ひとことで**：ブラウザのセキュリティ上、別ドメインの API を呼ぶにはサーバ側が「このフロントからの呼び出しを許可する」と宣言する必要があります。それが CORS です。

## 配置パターン（2 案と推奨）

| 案 | 内容 | 長所 | 短所 | ひとことで |
|----|------|------|------|------------|
| A. バックエンド集約 | プロバイダ取得〜タイル生成までサーバ | キー秘匿・キャッシュ統一 | サーバ維持が必要 | 裏方をまとめて持つ王道構成 |
| B. 静的ホスティングのみ | クライアントが直接プロバイダ | 運用が軽い | CORS・鍵・レート制限が不利 | サーバなしだが制約が多い |

**推奨（MVP）: 案 A の論理** — イングエスト＋キャッシュ＋配信を **Cloudflare Workers 上の単一アプリ**としてまとめ、早期マイクロサービス化はしない。**静的 UI は GitHub Pages に分離**（物理は `09`）。

**ひとことで**：MVP では「サーバ側で取ってキャッシュして出す」案 A を採り、ただしサーバは 1 つのアプリにまとめ、画面だけ Pages に分けます。

## 外部インターフェース

### クライアント向け（概要）

- **メタデータ API（JSON）必須**: クライアントは利用可能な時刻・鮮度・タイル URL の組み立てに必要な情報を**ここからのみ**取得する（時刻の推測禁止）。
- **タイル取得**: メタデータが返す `tile_url_template` と `frames[].id`（または同等のプレースホルダ）に従い GET する。

### 初心者向け: メタデータとは

アプリ（ブラウザ側）が毎回「いまどの雨のコマ（フレーム）があるか」「タイル画像の URL をどう組み立てるか」を**自分で当てにいかない**ようにするため、**サーバに聞く公式の答え**がメタデータです。

具体的には、クライアントはメタデータ API に問い合わせて、だいたい次のようなことを教えてもらいます。

- **どのフレーム（時系列のコマ）が存在するか**（`frames` の一覧）
- **そのそれぞれについて、地図のズーム min/max はどこまで保証するか**（`zoom_range`）
- **保証する地図の範囲はどこか**（`coverage`）
- **タイル 1 枚の URL をどう作るか**（`tile_url_template` に `{frame_id}`, `{z}`, `{x}`, `{y}` を当てはめる）

つまり **「いまサーバが提供している雨タイルのカタログ＋ URL の型」** が JSON で返ってくる契約だと考えるとよいです。クライアントはこの JSON を信頼してタイルを取りに行き、時刻やパスを勝手に推測しません。

### メタデータ API — 契約（規範）

以下は **v1** の論理スキーマである。URL パスは実装で `/api/v1/radar/meta` のように **バージョン付きプレフィックス**を付ける（MUST＝付けなければならない）。

**時刻表現**: すべて **UTC の ISO 8601**（例: `2026-05-06T12:34:56Z`）。クライアント側のローカル表示に変換する。

**必須フィールド（トップレベル）**

| フィールド | 型 | 意味 | ひとことで |
|------------|-----|------|------------|
| `contract_version` | string | 固定値 `"1"`（論理メジャーバージョン） | この JSON の「約束の版」 |
| `crs` | string | タイルが前提とする参照系（例: Web Mercator を `EPSG:3857` と明示） | 座標系の名前 |
| `tile_url_template` | string | `{frame_id}`, `{z}`, `{x}`, `{y}` のプレースホルダを含む **絶対 HTTPS URL**（本プロジェクトでは **API オリジン**＝Cloudflare Worker。`09`）。v1 では `{time}` ではなく `{frame_id}` を使う（MUST） | タイル URL のひな形（全部そろった HTTPS の文字列） |
| `latest_available_time` | string \| null | **観測・解析寄り**（`frames[].role === "analysis"`）のうち最も新しい `time`。無い場合は `null` | 「いま」に近い実況系コマの最新 |
| `stale` | boolean | 鮮度欠如（プロバイダ期待より遅い・連続失敗など）。算出規則は `radar-catalog` が担当（`04`） | データが古い／不安定なとき true |
| `coverage` | object | このメタデータでタイル取得を保証する地理範囲。クライアントの表示・タイル要求を絞るために使う | 保証する地図上の矩形など |
| `frames` | array | 利用可能フレームの一覧（時系列順・昇順 MUST） | 利用可能なコマのリスト（古い→新しい） |

**`frames[]` 各要素の必須フィールド**

| フィールド | 型 | 意味 | ひとことで |
|------------|-----|------|------------|
| `id` | string | タイル URL に埋め込むフレーム識別子（例: ISO 時刻文字列またはOpaque ID） | URL に入れる「このコマの ID」 |
| `time` | string | 当該フレームの参照時刻 valid（UTC ISO 8601）。ナウキャストでは予報コマもここに含む | そのコマの時刻 |
| `zoom_range` | object | `min`, `max` 整数。当該フレームについて **coverage 内のタイル GET が成功すべき**ズーム範囲 | このズーム幅なら取れるはず、という約束 |
| `role` | string \| 省略 | `"analysis"`（実況・解析寄り）または `"forecast"`（短期予報コマ）。省略は旧クライアント向け | コマの種別 |

**`coverage` の必須フィールド**

| フィールド | 型 | 意味 | ひとことで |
|------------|-----|------|------------|
| `bbox` | number[4] | `[west, south, east, north]`。座標は `EPSG:4326` の経緯度 | 保証範囲の四角（緯度経度） |
| `tile_matrix` | string | タイル座標系。v1 では `XYZ` を基本とする | タイルの数え方（XYZ 方式） |

`coverage` 外のタイルは公開保証対象外であり、サーバは短期キャッシュ可能な `404` または透明タイル等で応答してよい（MAY＝してもよい）。クライアントは `coverage.bbox` でレイヤ表示範囲を絞り、範囲外の大量リクエストを避ける（SHOULD＝した方がよい）。

**任意フィールド**

| フィールド | 型 | 意味 | ひとことで |
|------------|-----|------|------------|
| `time_estimated` | boolean | `latest_available_time` またはフレーム時刻が推定である場合 true | 時刻が推定値である印 |
| `default_frame_id` | string \| null | 主用途（直近の降水の見通し）向けに推奨する `frames[].id`。無い場合はクライアントが `latest_available_time` 等で決める | 初期表示の「おすすめ」コマ |
| `forecast_available` | boolean \| 省略 | メタに短期予報コマ（`role: "forecast"`）が含まれるか | N2 取得失敗時など false |
| `latest_analysis_time` | string \| null \| 省略 | `latest_available_time` と同義でよい。**観測・解析寄り**の最新 valid（UTC ISO）。明示的に分解したい場合に使用 | 解析最新の冗長表現（任意） |
| `provider_attribution` | string | 地図上に表示する帰属表示テキスト | データ元のクレジット文言 |

**エラー応答（規範）**

- `4xx` / `5xx` 時は **JSON オブジェクト**を返す（SHOULD）。フィールド: `error_code`（機械可読）, `message`（人間向け短文）。
- クライアントは **直前に取得済みのメタデータをそのまま表示し続けてよい（MAY）** が、**タイル URL をキャッシュして無期限に使い回してはならない（MUST NOT）**。再取得のバックオフは実装フェーズで定義する。

**ひとことで**：エラーのときも分かりやすい JSON を返すのが望ましい。画面は前のメタでしのいでもよいが、**タイルの URL だけを永久保存して使い続けるのは禁止**です（新しいフレームやパスに追従できなくなるため）。

**キャッシュ**

- メタデータ応答には適切な `Cache-Control` を付与する（SHOULD）。タイル応答はフレーム単位で `immutable` に近い長め TTL でもよいが、**新フレーム公開後は新しい `id`／パス**で区切る（`03` のキャッシュキー規約）。

### 参考: クライアントとメタデータの組み合わせ（informative）

**参考（informative）**＝読み物・例であり、規範そのものの代替ではありません。

メタデータの `frames[].id`、`coverage`、`zoom_range`、`tile_url_template` が、クライアントが発行するタイル GET の**唯一の入力**である。これにより、実装の分岐と範囲外タイルの無駄な取得を減らす。

### 運用向け

- **ヘルス**（例: `/healthz`）: プロセス生存、**最終成功フェッチ時刻**、クリティカル依存の疎通
- **メトリクス**: 取得失敗率、キャッシュヒット率、レンダリング遅延（実装フェーズで詳細化）

### ヘルスとメタデータの関係

- **ヘルス**はオペレータ向けの**狭義の生存・依存チェック**（「サービスは応答し、最近成功したパイプライン実行があるか」）を示す。
- **メタデータ**はエンドユーザー向けの**データ鮮度と利用可能フレーム**の真実である。
- 両者が矛盾してよい場合: ヘルスは OK でも **stale=true**（例: プロバイダは生きているが返却が古い）。逆にヘルス NG ならメタデータは 503 となりうる — このときクライアントは前述のとおり前回メタを表示してよい。

**ひとことで**：**ヘルス**は「機械は動いてる？」、**メタデータ**は「ユーザーに見せる雨は新しい？」という別の役割です。どちらか片方だけ見て判断しすぎない、という整理です。

## レビュー反映（rev.2）

- 「1 サービス」でも**プロセス内ではレイヤ境界を守る**（後述 `04-module-boundaries.md`）。早期マイクロサービス化はしない。
- メタデータ API を **必須** とし、クライアントが時刻を推測しない設計にした（シンプルさと保守性の両立）。

## レビュー反映（rev.3）

- メタデータの **フィールドレベル契約**・エラー・キャッシュ・時刻 TZ を追加した。
- ヘルスと **stale** の役割分担を固定した。

## デプロイ反映（rev.7）

- **GitHub Pages ＋ Cloudflare Workers** を規範配置とし、`tile_url_template` の **絶対 URL 化**と **CORS** を明記（`09`）。

## 最適化反映（rev.8）

- `tile_url_template` のプレースホルダを **`{frame_id}` / `{z}` / `{x}` / `{y}`** に統一し、クライアントが時刻表現に依存しない契約へ整理した。
- メタデータに **`coverage`** を追加し、公開保証対象を「coverage 内かつ `zoom_range` 内」のタイルに限定した。これにより、全世界タイルを暗黙に保証する誤解と、範囲外リクエストによる無駄な負荷を避ける。

# 設計レビュー（批判的）と rev.3 への反映

## レビュー実施方針

サブエージェントに **保守性・シンプル性・処理タイミング設計・責務の独立性** の4観点のみで批判的レビューを実施した。以下は要約であり、対応は `design/*.md` の **rev.3** に反映済み。

## エグゼクティブサマリ（レビュー側）

- メタデータ API を「必須」としつつ、**フィールドスキーマ・エラーモデル・バージョン**が未定義だと契約テストとクライアント実装が分岐する。
- **公開順序**（メタデータとタイルの整合、`T_pub` の定義）が未定義だと、部分ズームや CDN 無効化で不整合が起きやすい。
- ポーリング「0.5〜1倍」は**基準が曖昧**で、過負荷・陳腐化の両リスクがある。
- パススルータイル時の **`TileStore` 書き込み責務**が曖昧だと ingest と adapter の境界が崩れる。
- フェイクプロバイダは**環境分離ポリシー**を要件レベルで書かないと設定ミスリスクが残る。

## 修正対応マップ（must-fix → 反映先）
